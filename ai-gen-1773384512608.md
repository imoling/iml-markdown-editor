# 会议助手技术架构设计文档

## 1\. 项目概述

### 1.1 项目背景

本项目旨在开发一个基于 OpenAI Whisper 模型的智能会议助手系统。该系统能够实时或异步地处理会议音频，将其转换为高精度的文本记录，并在此基础上提供智能摘要、关键信息提取、任务项识别等功能，以提升会议效率和信息留存价值。

### 1.2 核心目标

1.  **高精度语音转文本**：利用 Whisper 模型实现多语言、带标点的会议录音转写。
    
2.  **实时/异步处理**：支持会议过程中的实时字幕生成与会议结束后的完整录音文件处理两种模式。
    
3.  **后处理与智能分析**：对转写文本进行智能处理，生成会议纪要、提取行动项、关键决策点等。
    
4.  **可扩展与可维护**：设计松耦合、模块化的架构，便于未来集成更多AI能力或适配不同部署环境。
    

### 1.3 非功能性需求

*   **性能**：异步处理时，对于1小时音频，转写时间应控制在音频时长的0.3-0.5倍（即18-30分钟）内。实时模式延迟应低于5秒。
    
*   **准确性**：在普通话/标准英语、中等质量会议室录音环境下，转写词错误率（WER）目标低于10%。
    
*   **可用性**：系统核心服务可用性目标不低于99.5%。
    
*   **安全性**：确保音频及转写文本数据在传输和存储过程中的加密与访问控制。
    

## 2\. 总体架构设计

本系统采用分层、模块化的微服务架构，将核心功能解耦，便于独立开发、部署和扩展。

### 2.1 架构图

```
[客户端] --> (API Gateway) --> [微服务集群] --> [数据存储与AI模型]
```

*   **客户端**：Web前端、移动端、第三方系统集成接口。
    
*   **API网关**：统一的请求入口，负责路由、认证、限流、日志。
    
*   **微服务集群**：包含音频处理、文本分析、用户管理等独立服务。
    
*   **数据与模型层**：数据库、对象存储、Whisper模型服务。
    

### 2.2 核心组件

1.  **API网关**
    
2.  **音频摄取服务**
    
3.  **Whisper模型推理服务**
    
4.  **文本后处理与分析服务**
    
5.  **用户与会话管理服务**
    
6.  **存储服务**
    

## 3\. 详细组件设计

### 3.1 API网关

*   **技术选型**：Nginx / Kong / Spring Cloud Gateway。
    
*   **职责**：
    
    *   客户端请求路由至对应微服务。
        
    *   JWT令牌验证与用户身份鉴权。
        
    *   速率限制与防刷。
        
    *   访问日志收集。
        
    *   SSL/TLS终止。
        

### 3.2 音频摄取服务

*   **职责**：接收、验证、预处理并暂存客户端上传的音频流或文件。
    
*   **流程**：
    
    1.  接收上传（支持WebSocket流式上传及HTTP文件上传）。
        
    2.  格式验证与统一转换（如统一转为16kHz, 16-bit PCM的WAV格式）。
        
    3.  音频切片（针对长音频，便于分批推理和并行处理）。
        
    4.  将预处理后的音频片段存入**消息队列**，并生成任务元数据。
        
*   **技术栈**：Python (FastAPI/Flask)， FFmpeg， Apache Kafka / RabbitMQ。
    

### 3.3 Whisper模型推理服务

*   **职责**：从消息队列消费音频任务，调用Whisper模型进行语音识别，输出原始文本。
    
*   **部署模式**：
    
    *   **方案A（容器化）**：将Whisper模型封装为gRPC或HTTP API服务，使用TensorFlow Serving或TorchServe。
        
    *   **方案B（直接调用）**：在服务内直接加载PyTorch/TensorFlow模型。
        
*   **优化考虑**：
    
    *   **模型选择**：根据精度与速度权衡，选择 `base`、`small` 或 `medium` 尺寸模型。
        
    *   **硬件加速**：部署在配备GPU（NVIDIA）的服务器上，利用CUDA加速推理。
        
    *   **批处理**：对短音频切片进行批处理推理，提升GPU利用率。
        
    *   **动态加载**：支持多语言检测或指定语言模型。
        
*   **输出**：将结构化的转写结果（包含时间戳、文本、置信度）写入数据库并发布到消息队列，供下游服务消费。
    
*   **技术栈**：Python， PyTorch/TensorFlow， ONNX Runtime (可选)， CUDA。
    

### 3.4 文本后处理与分析服务

*   **职责**：对Whisper输出的原始文本进行加工和深度分析。
    
*   **功能模块**：
    
    1.  **文本后处理**：标点修正、数字规范化、去除语气词、分段（基于静默检测或语义）。
        
    2.  **说话人分离**（可选增强）：集成如pyannote.audio等工具，区分不同发言人（需额外标注数据或声纹信息）。
        
    3.  **智能分析**：
        
        *   **摘要生成**：利用TextRank或微调的Transformer模型（如BART、T5）生成会议摘要。
            
        *   **关键信息提取**：使用NER技术提取人名、组织、时间、产品等实体。
            
        *   **行动项识别**：基于规则模板或分类模型识别会议中提出的任务（如“小明下周提交方案”）。
            
        *   **主题聚类**：对会议内容进行主题划分。
            
*   **技术栈**：Python， spaCy/NLTK， Hugging Face Transformers。
    

### 3.5 用户与会话管理服务

*   **职责**：管理用户信息、会议（会话）元数据、权限控制。
    
*   **数据模型**：
    
    *   `User`: 用户基本信息、权限。
        
    *   `MeetingSession`: 会议唯一标识、创建时间、状态（处理中/完成）、关联用户、设置（如语言）。
        
    *   `Transcript`: 与会议关联的完整转写文本、分析结果（摘要、行动项等）。
        
*   **技术栈**：Node.js/Java/Python (任一Web框架)， 关系型数据库（如PostgreSQL）。
    

### 3.6 存储服务

*   **对象存储**（如MinIO, AWS S3, 阿里云OSS）：
    
    *   存储原始音频文件。
        
    *   存储处理后的音频片段。
        
*   **关系型数据库**（如PostgreSQL）：
    
    *   存储用户、会议元数据、文本分析结果的结构化数据。
        
*   **向量数据库**（可选，如Milvus, Pinecone）：
    
    *   存储会议文本的嵌入向量，用于实现语义搜索、相似会议推荐等高级功能。
        
*   **缓存**（如Redis）：
    
    *   缓存高频访问的会议结果。
        
    *   存储实时会议的字幕流状态。
        

## 4\. 数据处理流程

### 4.1 异步处理流程（会后上传）

1.  用户通过客户端上传完整会议录音文件。
    
2.  API网关验证后，请求抵达**音频摄取服务**。
    
3.  该服务预处理音频，切片后向消息队列发布`AudioProcessingTask`。
    
4.  **Whisper模型推理服务**监听队列，取任务并推理，发布`TranscriptReadyEvent`。
    
5.  **文本后处理与分析服务**监听事件，进行文本加工与分析，将最终结果保存至数据库。
    
6.  用户可通过客户端查询处理状态并查看最终会议纪要。
    

### 4.2 实时处理流程（会中字幕）

1.  客户端通过WebSocket建立与**音频摄取服务**的长连接，实时推送音频流。
    
2.  摄取服务将流式音频缓冲并切成短片段（如每2秒），实时发布至消息队列。
    
3.  **Whisper模型推理服务**以更高优先级消费这些短片段任务，快速返回带时间戳的文本。
    
4.  推理结果通过WebSocket或Server-Sent Events (SSE)实时推回给客户端，显示为实时字幕。
    
5.  会议结束后，系统自动触发一个合并与后处理任务，生成完整的会议记录。
    

## 5\. 部署与运维架构

### 5.1 基础设施

*   **云原生部署**：建议使用Docker容器化所有微服务，通过Kubernetes进行编排管理，实现弹性伸缩、自愈和滚动更新。
    
*   **混合部署**：将计算密集的**Whisper模型推理服务**部署在GPU节点上，其他服务部署在CPU节点。
    

### 5.2 监控与日志

*   **应用监控**：集成Prometheus收集各服务指标（请求量、延迟、错误率、GPU利用率）。
    
*   **日志聚合**：使用ELK Stack（Elasticsearch, Logstash, Kibana）或Loki收集、索引和查询分布式日志。
    
*   **链路追踪**：使用Jaeger或Zipkin实现请求在微服务间的全链路跟踪，便于故障定位。
    

### 5.3 安全设计

*   **传输安全**：全程使用HTTPS/WSS。
    
*   **认证授权**：基于JWT的Token认证，OAuth 2.0支持。
    
*   **数据加密**：静态数据（存储中）使用AES-256加密。动态数据（传输中）使用TLS 1.3。
    
*   **隐私考虑**：提供音频数据自动过期删除策略。敏感信息在分析后可选择脱敏。
    

## 6\. 扩展性与未来演进

1.  **多模型支持**：架构设计允许接入其他ASR模型（如Google Speech-to-Text, 阿里云语音识别）作为备选或对比。
    
2.  **功能插件化**：文本分析模块可设计为插件体系，方便新增“情感分析”、“争议点检测”等功能。
    
3.  **离线部署包**：为满足私有化部署需求，可提供包含所有依赖的Docker Compose或Helm Chart包。
    
4.  **集成能力**：提供标准API，便于与Teams、Zoom、钉钉等会议系统，以及Notion、Confluence等知识库系统集成。
